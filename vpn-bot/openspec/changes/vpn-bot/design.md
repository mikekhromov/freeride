## Context

Репозиторий **vpn-bot** — один бинарник на Go 1.22: Telegram-бот (`gopkg.in/telebot.v3`), локальная БД **SQLite** (`modernc.org/sqlite`), интеграция с **Hiddify** по HTTP для создания пользователей и выдачи ссылок. Админы задаются через `ADMIN_IDS`; пользователи проходят путь «запрос → pending → одобрение → active с UUID в Hiddify». Деплой типично через systemd (`vpn-bot.service`) и переменные окружения.

Новый объём change добавляет UX-доставку нескольких ссылок подключения, визуальные карточки с брендингом и self-service команды пользователя.

## Goals / Non-Goals

**Goals:**

- Зафиксировать границы ответственности слоёв: handlers → store/services → внешние API.
- Описать жизненный цикл записи `users` и согласованность с Hiddify при approve/revoke.
- Задать ожидаемые режимы работы бота: long polling vs webhook (конфигурация без смены бизнес-логики).
- Уточнить формат Telegram Proxy ссылки: домен 4-го уровня с префиксом `users.` вместо прямого IP.
- Разделить выдачу VPN-подключений на 3 типа ссылок: WireGuard, Full Xray, all configs.
- Добавить кнопки скачивания для WireGuard и Full Xray с выдачей локально сформированных `.txt`-конфигов.
- Добавить self-service команды пользователя: перевыпуск конфигов и просмотр минимальной статистики трафика.
- Добавить отправку карточек с подписью "VPN" и "Telegram Proxy" с использованием `golang.org/x/image`.

**Non-Goals:**

- Переписывание стека или смена Telegram-библиотеки.
- Детальное описание конфигурации nginx/SSL (есть пример `nginx-bot.conf`, но не часть контракта спек).
- Миграция на другую панель VPN вместо Hiddify.
- Полноценная BI-аналитика трафика; требуется только минимальная user-facing статистика.

## Decisions

1. **Единый источник состояния пользователя — SQLite.** Hiddify хранит технического пользователя и UUID; бот хранит `telegram_id`, опционально `telegram_username`, `status`, `hiddify_uuid`, метаданные одобрения. При активации сначала создаётся пользователь в Hiddify, затем атомарно (в рамках одной транзакции БД в коде сейчас — через последовательные вызовы) обновляется строка пользователя; при сбое после создания в Hiddify возможен orphan — риск зафиксирован ниже.

2. **Статусы пользователя:** как минимум `pending`, `active`, `banned` — семантика определяется store и хендлерами; отклонение запроса переводит в `banned` (как в текущем коде).

3. **Идентификация админа:** только по числовому Telegram user id из множества `ADMIN_IDS`; команды админа не доступны остальным.

4. **Таймауты контекста:** отдельные таймауты для статистики, approve и callback-операций — снижают зависание при медленном Hiddify.

5. **Доставка подключений в два шага:** при успешной активации/инициализации бот отправляет:
   - сообщение №1: картинка "VPN" + три ссылки (WireGuard, Full Xray, все конфиги),
   - сообщение №2: картинка "Telegram Proxy" + proxy-ссылка.
   После этого отправляется финальное текстовое сообщение с тегом поддержки.

6. **Формирование Telegram Proxy URL:** URL должен использовать host, начинающийся с `users.` (например `users.<env>.<domain>`), даже если backend возвращает IP-адрес.

7. **Генератор карточек:** отдельный утилитарный модуль рендера изображений на базе `golang.org/x/image` поверх брендового фона (из assets), с коротким заголовком (ровно "VPN" или "Telegram Proxy").

8. **Self-service команды пользователя:**
   - `/revoke` (и alias `/revok`) инициирует забывание/перевыпуск собственных конфигов с проверкой, что пользователь активен.
   - `/stats` возвращает краткую статистику потребления (минимум: использовано, лимит, остаток/процент).

9. **Скачивание конфигов через callback-кнопки:**
   - В VPN-сообщении добавляются кнопки "Скачать WireGuard" и "Скачать Full Xray".
   - По нажатию бот выполняет server-side загрузку содержимого по соответствующей URL-ссылке и отправляет документ пользователю.
   - Имя файла нормализуется по шаблону `<username-or-id>_<protocol>.txt` (например `mike_wireguard.txt`, `123456_xray.txt`).

## Risks / Trade-offs

- **[Hiddify недоступен при approve]** → пользователь остаётся в неоднозначном состоянии; операция должна завершиться ошибкой в Telegram без частичной активации в БД (текущая логика: ошибка до `ActivateUser`).
- **[Пользователь создан в Hiddify, запись в БД не обновилась]** → «осиротевший» пользователь в панели; mitigation: ручная очистка / будущая задача на компенсацию (вне этого change).
- **Reject переводит в `banned`** → пользователь не может повторно запросить доступ без вмешательства админа; это осознанное поведение кода, не баг.
- **[Рендер карточек увеличивает сложность деплоя]** → требуется хранение фонового asset-файла и fallback на текстовый ответ при ошибке рендера.
- **[Несовпадение домена `users.*` и реального TLS/роутинга]** → валидация формата через конфиг и smoke-test ссылок в staging обязательны.
- **[Удаленная ссылка недоступна в момент скачивания]** → пользователю отправляется дружелюбная ошибка и повторная кнопка/инструкция, без утечки технических деталей.

## Migration Plan

Порядок выката:
1. Добавить переменные окружения для домена прокси и тега поддержки.
2. Добавить image-assets в deploy bundle.
3. Выпустить бинарник с новыми командами и генерацией карточек.
4. На стейдже проверить end-to-end: approve -> 2 карточки + ссылки + финальный support-tag, затем self `/stats` и self `/revoke`.
4.1 Проверить, что кнопки скачивания WireGuard/Xray отдают корректные `.txt` файлы с ожидаемыми именами.
5. Подготовить rollback: отключить новые сообщения/команды feature-flag или вернуть предыдущий бинарник.

## Open Questions

- Нужен ли отдельный статус «rejected» вместо `banned` для отклонённых заявок (продуктовое решение, сейчас в коде — `banned`).
- Нужна ли идемпотентность повторного `pending` для ранее `banned` пользователя (сейчас `SetPendingRequest` не переводит `banned` в `pending`).
- Какой exact-format должен быть у пользовательской статистики (`/stats`): только текст или текст + единицы в GB/MB.
- Какой точный тег поддержки использовать в проде (значение должно быть задано конфигом).
- Нужна ли явная кнопка повторной попытки скачивания, если источник временно недоступен.
