# Proxy Share Bot Agent Spec

## Контекст

Агент работает над системой Telegram-бота, которая управляет выдачей прокси-доступов через шаринг-ссылки.

Система **не пытается запретить физическую пересылку уже выданных proxy credentials**.  
Система **контролирует только официальный канал выдачи доступов через бота**.

Основная бизнес-идея:

- пользователь приходит в бота
- может создать шаринг-ссылку
- шаринг-ссылка позволяет выдать доступ максимум `N` пользователям
- по умолчанию `N = 5`
- каждый новый пользователь должен прийти **через бота**
- только бот принимает решение, выдать доступ или отказать
- при превышении лимита новый пользователь не получает доступ
- владельцу ссылки отправляется уведомление о превышении лимита

---

## Основные цели

1. Реализовать **контролируемую выдачу proxy credentials** через Telegram-бота.
2. Ограничить количество официальных активаций одной шаринг-ссылки.
3. Позволить владельцу отслеживать использование его ссылки.
4. Позволить отзывать шаринг-ссылки и запрещать новые активации.
5. Исключить сценарий, в котором 6-й пользователь получает доступ при лимите `5`.
6. Сделать логику простой, прозрачной и проверяемой на уровне БД и backend.

---

## Что система делает

Система должна:

- хранить Telegram-пользователей
- хранить шаринг-ссылки
- хранить факты активации ссылок
- проверять лимит активаций
- выдавать proxy credential только после успешной активации
- отказывать при исчерпанном лимите
- уведомлять владельца ссылки о попытке превышения лимита
- поддерживать revoke шаринг-ссылки

---

## Что система НЕ делает

Система не должна:

- считать, что proxy credential нельзя переслать другому человеку
- обещать полную защиту от несанкционированного шаринга уже выданных настроек прокси
- привязывать Telegram proxy protocol к Telegram user identity на уровне transport layer
- блокировать 6-го пользователя на уровне прокси, если он уже получил готовые настройки в обход бота
- использовать Redis без явной необходимости
- усложнять MVP дополнительными слоями вроде очередей, event bus или распределенных блокировок

---

## Архитектурные принципы

### 1. Control plane и data plane разделены

Необходимо различать:

- **control plane** — бот, backend, БД, логика лимитов, активаций и revoke
- **data plane** — сам прокси и выданные credentials

Все решения о выдаче доступа принимаются **только в control plane**.

### 2. Ограничивается не proxy-link, а выдача доступов

Система не должна трактовать одну proxy-ссылку как объект, который сам умеет ограничивать число пользователей.

Правильная модель:

- одна **share link**
- максимум `N` успешных активаций
- в текущем проекте используется **только Group mode**: всем успешно активировавшимся выдаётся **один общий credential** на эту ссылку (см. раздел «Режимы выдачи credentials»)

### 3. Блокировка 6-го пользователя происходит до выдачи доступа

Если лимит исчерпан:

- новая активация не создается
- proxy credential не выдается
- владельцу отправляется уведомление

### 4. Источником истины являются факты активации

Количество использований ссылки определяется через число записей активаций, а не через доверие к вычисленному полю без проверки.

### 5. Проверка лимита должна быть атомарной

Активация ссылки обязана происходить в транзакции, чтобы два конкурентных запроса не заняли один и тот же последний слот.

## Сущности

### TgUser

Представляет Telegram-пользователя, взаимодействующего с ботом.

Поля:

- `id`
- `telegram_id`
- `username`
- `first_name`
- `status`
- `created_at`

Правила:

- `telegram_id` уникален
- пользователь создается или обновляется при первом взаимодействии с ботом

---

### ShareLink

Представляет шаринг-ссылку, которую пользователь создает для приглашения других.

Поля:

- `id`
- `owner_user_id`
- `token`
- `max_uses`
- `status`
- `created_at`
- `revoked_at` nullable

Правила:

- `token` уникален
- `owner_user_id` обязателен
- `max_uses` по умолчанию равен `5`
- допустимые статусы:
  - `active`
  - `revoked`
  - `exhausted`

---

### ShareLinkActivation

Представляет факт успешной активации шаринг-ссылки конкретным пользователем.

Поля:

- `id`
- `share_link_id`
- `activated_user_id`
- `created_at`

Правила:

- один и тот же пользователь не может дважды активировать одну и ту же ссылку
- уникальность обеспечивается ограничением:
  - `unique (share_link_id, activated_user_id)`

---

### ProxyCredential

Представляет credential, выданный после успешной активации.

Поля зависят от реализации прокси, но модель должна быть универсальной.

Поля:

- `id`
- `share_link_id`
- `user_id`
- `credential_type`
- `host`
- `port`
- `login` nullable
- `password` nullable
- `secret` nullable
- `status`
- `issued_at`
- `revoked_at` nullable
- `expires_at` nullable

Статусы:

- `active`
- `revoked`
- `expired`

Правила:

- credential выдается только после успешной активации
- в **Group mode** (единственный режим в этом проекте) на одну `share_link` существует **одна** запись группового credential; поле `user_id` при необходимости трактуется как владелец credential при создании или остаётся вспомогательным — конкретная схема фиксируется в миграциях, но **отдельный credential на каждого участника не создаётся**

---

## Режимы выдачи credentials

### Зафиксированное решение: только Group mode (режим A)

**Проект реализует исключительно Group mode.** Режим B (per-user credentials) в текущий план **не входит** и не должен усложнять код ветвлениями «на будущее».

Все пользователи, пришедшие по одной шаринг-ссылке, получают **один и тот же** credential группы (данные подключения совпадают у всех активированных участников).

Инварианты:

- одна `share_link`
- один групповой `ProxyCredential` (или эквивалентная одна запись с параметрами подключения), создаваемый при первой успешной активации или заранее привязанный к ссылке — согласовать с реализацией, главное: **не N отдельных credential на N пользователей**
- максимум `N` участников с доступом через бота
- 6-й и далее не получают credential через бота

Риски (принимаются осознанно):

- любой участник группы может переслать общий credential другим людям
- revoke ссылки по смыслу затрагивает **весь** общий доступ по этой ссылке

---

### Режим B. Per-user mode (вне scope)

Отдельный credential на каждого активировавшего пользователя **не реализуется** в рамках текущего плана. Возможное развитие — см. «Будущие расширения».

---

## Основные инварианты

Агент обязан сохранять следующие инварианты:

1. У одной шаринг-ссылки не может быть больше `max_uses` успешных активаций.
2. Если число активаций равно `max_uses`, новая активация запрещена.
3. 6-й пользователь при лимите `5` не должен получить credential.
4. Credential не выдается без записи успешной активации.
5. Revoked share link не может быть активирована повторно.
6. Повторный запуск `/start <token>` тем же пользователем не должен занимать новый слот.
7. Проверка лимита и создание активации должны быть атомарными.
8. Уведомление владельцу отправляется при попытке активации сверх лимита.
9. Система не должна обещать невозможность пересылки уже выданных credentials.
10. Источником истины для лимита являются активации, а не клиентское состояние.

## Поведение бота

### Сценарий: создание шаринг-ссылки

Когда пользователь запрашивает создание шаринг-ссылки, система должна:

1. убедиться, что пользователь зарегистрирован в БД
2. создать новую запись `ShareLink`
3. сгенерировать уникальный `token`
4. установить `max_uses = 5`, если не передано иное
5. вернуть пользователю ссылку вида:

`https://t.me/<BOT_NAME>?start=<token>`

---

### Сценарий: активация ссылки новым пользователем

Когда пользователь открывает `/start <token>`, система должна:

1. найти `ShareLink` по `token`
2. проверить, что ссылка существует
3. проверить, что статус ссылки `active`
4. проверить, не активировал ли этот пользователь ссылку ранее
5. в транзакции посчитать текущее число активаций
6. сравнить число активаций с `max_uses`
7. если лимит не исчерпан:
   - создать `ShareLinkActivation`
   - в **Group mode**: если групповой `ProxyCredential` для этой `share_link` ещё не создан — создать одну запись; иначе использовать существующий групповой credential
   - вернуть пользователю те же данные для подключения, что и у остальных участников группы
8. если лимит исчерпан:
   - не создавать `ShareLinkActivation`
   - не выдавать `ProxyCredential`
   - уведомить владельца
   - показать пользователю сообщение об отказе

---

### Сценарий: повторный заход пользователя по своей же ссылке

Если пользователь уже активировал ссылку ранее, система должна:

- не занимать новый слот
- не создавать повторную активацию
- вернуть уже существующий credential либо переотправить его, если это допустимо логикой продукта

---

### Сценарий: превышение лимита

Если активаций уже `5` из `5`, а приходит новый пользователь, система должна:

- отказать в активации
- не выдавать credential
- уведомить владельца о попытке превышения лимита

Пример пользовательского ответа:

- `Лимит подключений по этой ссылке исчерпан.`

Пример уведомления владельцу:

- `По вашей ссылке была попытка подключения сверх лимита. Текущий лимит: 5 из 5.`

---

### Сценарий: revoke ссылки

Когда владелец или администратор отзывает шаринг-ссылку, система должна:

- изменить статус `ShareLink` на `revoked`
- запретить все новые активации

Поведение уже выданных credentials определяется проектным режимом:

- в MVP допускается не отзывать уже выданные credentials автоматически
- если включен строгий режим, revoke может дополнительно отзывать связанные credentials

Агент не должен менять это поведение неявно. Оно должно быть явной настройкой системы.

---

## Требования к транзакционности

Агент должен реализовывать активацию ссылки так, чтобы не было race condition.

Обязательное требование:

- выборка `ShareLink` для активации должна происходить с блокировкой строки
- подсчет текущих активаций и вставка новой активации должны быть в одной транзакции

Допустимый псевдопорядок:

1. `BEGIN`
2. `SELECT share_link FOR UPDATE`
3. `COUNT activations`
4. если мест нет — `ROLLBACK`
5. если место есть — `INSERT activation`
6. `COMMIT`

---

## Требования к БД

Для MVP агент должен использовать **одну PostgreSQL базу данных**.

Агент не должен добавлять Redis, если нет отдельного требования на:

- rate limit
- кэширование
- TTL tokens вне PostgreSQL
- очереди
- распределенные блокировки

Минимальные ограничения БД:

- уникальность `tg_users.telegram_id`
- уникальность `share_links.token`
- уникальность `(share_link_id, activated_user_id)` в `share_link_activations`

---

## Инфраструктура: Docker, база данных и Telegram MTProto proxy

Агент и развёртывание опираются на **Docker Compose** (или эквивалент) с **одним сервисом PostgreSQL** и **одним сервисом приложения** (бот + backend). Отдельно описывается **установка MTProto proxy** (Telegram proxy) как часть data plane. Redis и прочие СУБД в MVP не добавляются без отдельного требования.

### Сервисы Compose (минимум)

| Сервис | Назначение |
|--------|------------|
| `db` | PostgreSQL, персистентное хранилище |
| `app` | процесс Telegram-бота (и HTTP API, если появится) |

**Telegram MTProto proxy** (см. «Установка и параметры Telegram MTProto proxy» ниже) — часть **data plane**: может быть в том же Compose на стенде или на **отдельном хосте** в продакшене; боту для выдачи нужны только согласованные `host` / `port` / `secret`, которые попадают в `ProxyCredential`.

Имена сервисов и сеть Compose могут отличаться; важно, чтобы приложение подключалось к хосту БД по **внутреннему DNS-имени сервиса** (например `db:5432`), а не через `localhost` изнутри контейнера приложения.

### Параметры образа PostgreSQL (контейнер `db`)

Переменные окружения для официального образа `postgres` (значения — **секреты**, не коммитить в репозиторий в открытом виде; для локали — `.env`, для продакшена — secrets orchestrator):

| Переменная | Назначение | Пример для локальной разработки (не копировать в прод как есть) |
|------------|------------|-------------------------------------------------------------------|
| `POSTGRES_USER` | пользователь БД | `freeride` |
| `POSTGRES_PASSWORD` | пароль пользователя | задаётся в `.env` |
| `POSTGRES_DB` | имя базы по умолчанию | `freeride` |
| `PGDATA` | каталог данных внутри контейнера | по умолчанию `/var/lib/postgresql/data`; менять только при необходимости |

Порт наружу:

- **локально**: проброс `5432:5432` удобен для отладки с хоста (DBeaver, `psql`).
- **продакшен**: порт БД наружу не публикуется; доступ только из сети приложения.

Том (volume):

- именованный том на `PGDATA`, чтобы данные переживали пересоздание контейнера, например `postgres_data:/var/lib/postgresql/data`.

### Параметры подключения приложения к БД

Приложение читает настройки из окружения. Допустимы два эквивалентных стиля (выбрать один и придерживаться его в коде и документации):

**Вариант 1 — одна строка:**

| Переменная | Назначение |
|------------|------------|
| `DATABASE_URL` | URL подключения PostgreSQL, например `postgresql://USER:PASSWORD@db:5432/freeride` |

**Вариант 2 — раздельно:**

| Переменная | Назначение |
|------------|------------|
| `DB_HOST` | хост БД (`db` в Compose) |
| `DB_PORT` | порт (обычно `5432`) |
| `DB_NAME` | имя базы (= `POSTGRES_DB`) |
| `DB_USER` | пользователь (= `POSTGRES_USER`) |
| `DB_PASSWORD` | пароль (= `POSTGRES_PASSWORD`) |

Дополнительно для продакшена при необходимости: `DB_SSLMODE` / параметры SSL в `DATABASE_URL`.

### Параметры контейнера приложения (`app`)

Минимум для бота и миграций (имена условные, зафиксировать в `.env.example`):

| Переменная | Назначение |
|------------|------------|
| `TELEGRAM_BOT_TOKEN` | токен бота от BotFather |
| `DATABASE_URL` или набор `DB_*` | подключение к PostgreSQL (см. выше) |
| `BOT_PUBLIC_NAME` | username бота без `@`, для ссылок `https://t.me/<BOT_PUBLIC_NAME>?start=<token>` |
| `MTPROXY_PUBLIC_HOST`, `MTPROXY_PORT`, `MTPROXY_SECRET` | параметры **MTProto proxy** для подстановки в выдаваемый credential (если хранятся в env, а не только в БД); см. «Установка и параметры Telegram MTProto proxy» |

При необходимости: `LOG_LEVEL`, `NODE_ENV` / `APP_ENV`, часовой пояс.

### Зависимости при старте

- приложение должно **ждать готовности PostgreSQL** (healthcheck у сервиса `db` + `depends_on` с условием `service_healthy`, либо скрипт ожидания порта — на усмотрение реализации, главное — не падать миграциями на «БД ещё не поднялась»).

### Установка и параметры Telegram MTProto proxy (data plane)

Под **Telegram proxy** здесь понимается **MTProto proxy** — сервер, к которому подключаются клиенты Telegram (мобильное приложение / десктоп) по протоколу MTProto. Это **не** SOCKS5 и **не** HTTP-прокси для браузера. Выдаваемые ботом `host` / `port` / `secret` должны соответствовать именно этому типу прокси.

Связь с продуктом:

- бот ограничивает только **официальную выдачу** строки подключения (credential);
- сам прокси-сервер **не обязан** понимать лимиты шаринг-ссылок — лимит обеспечивается в control plane (см. раздел «Что система НЕ делает»).

#### Варианты развёртывания

1. **Официальный образ / репозиторий MTProxy** — сборка из исходников или готовый Docker-образ от Telegram (актуальное имя образа и теги смотреть в [репозитории MTProxy](https://github.com/TelegramMessenger/MTProxy); не фиксировать в спеке устаревший тег без проверки).
2. **Отдельный VPS** — типичный продакшен: прокси слушает публичный IP, бот и БД на другом хосте; в `ProxyCredential` хранится **публичный** хост и порт прокси.
3. **Тот же Docker Compose, что и бот** — допустимо для разработки и малых установок; наружу пробрасывается только порт прокси (часто `443` TCP), не путать с портом PostgreSQL.

#### Параметры, которые нужны для credential и клиента

Их же бот сохраняет в сущности `ProxyCredential` и отображает пользователю (в Group mode — один набор на всю группу):

| Параметр | Назначение |
|----------|------------|
| Публичный **хост** | IP или доменное имя сервера, куда клиент Telegram подключается |
| **Порт** | TCP-порт прокси (часто `443`; должен быть открыт в firewall и проброшен в Docker, если используется контейнер) |
| **Secret** | секретный ключ MTProto (в клиенте часто вводится как hex-строка или через ссылку `tg://proxy?...`); длина и формат — по документации выбранного MTProxy |

Опционально у конкретных сборок MTProxy:

- **fake-tls** / режим маскировки под TLS — если поддерживается образом, параметры задаются согласно документации образа;
- **promotion tag** (рекламный тег Telegram) — если используется, задаётся при запуске прокси; на логику бота не влияет, только на отображение в клиенте.

#### Переменные окружения / аргументы запуска (ориентир)

Конкретные флаги зависят от образа; агент при внедрении сверяется с README выбранного `Dockerfile`. Обычно задаются:

| Параметр | Смысл |
|----------|--------|
| Секрет | `-s` / `SECRET` / аналог — **один** секрет на инстанс (или ротация по политике инфраструктуры; для MVP достаточно одного секрета на групповой credential) |
| Порт | `-p` / проброс `443:443` (или другой открытый порт) |
| Workers | число воркеров под нагрузку (например `-w` / переменная образа) |

Секрет и, при необходимости, домен/IP **не** должны хардкодиться в коде бота: для выдачи пользователям они берутся из конфигурации окружения (`MTPROXY_PUBLIC_HOST`, `MTPROXY_PORT`, `MTPROXY_SECRET` и т. п.) или из БД после первичной настройки — способ хранения выбирается в реализации, но **один источник правды** для строки подключения.

#### Проверка после установки

- с сервера: `nc -zv <HOST> <PORT>` или аналог — порт доступен;
- в Telegram: «Настройки → Данные и память → Использование прокси» — добавить MTProto proxy с теми же host/port/secret, что выдаёт бот.

#### Связь с ботом

- При создании группового credential бот подставляет параметры MTProto, согласованные с развёрнутым прокси (из env или БД).
- Смена секрета или хоста на стороне прокси требует обновления выданных пользователям данных или отдельной политики (ротация — вне MVP, если не оговорено).

## SQL-инварианты

Агент должен стремиться обеспечивать бизнес-правила не только кодом, но и ограничениями БД.

Минимальный набор:

- `telegram_id` unique
- `token` unique
- `(share_link_id, activated_user_id)` unique
- foreign keys между пользователями, ссылками и активациями обязательны

## Правила для кода

Агент должен:

- писать простой и прозрачный backend-код
- избегать лишних абстракций в MVP
- разделять domain logic, storage layer и bot handlers
- выносить проверку лимита в отдельный use case / service
- писать типы явно
- не использовать `as`, если можно обойтись корректной типизацией
- для комментариев в JavaScript использовать docjs-нотацию
- не хардкодить бизнес-лимиты в нескольких местах
- хранить `max_uses` в сущности `ShareLink`

Агент не должен:

- смешивать Telegram bot handler и SQL-логику в одну большую функцию
- принимать решение о лимите на клиенте
- доверять данным из callback payload как источнику истины
- создавать credential до успешной транзакционной активации
- рассчитывать, что прокси сам ограничит 6-го пользователя

---

## Правила принятия решений

Если есть выбор между двумя вариантами, агент должен руководствоваться следующими приоритетами:

1. Сначала корректность лимита
2. Потом транзакционная безопасность
3. Потом простота MVP
4. Потом расширяемость
5. Потом удобство UX

Если есть конфликт между “красивой архитектурой” и “надежным ограничением 6-го пользователя”, агент обязан выбрать надежное ограничение.

---

## Явные ограничения продукта

Агент обязан помнить и не нарушать следующие продуктовые ограничения:

1. Система контролирует только выдачу через бота.
2. Уже выданные proxy credentials могут быть пересланы вручную.
3. Полный антишаринг на уровне Telegram proxy невозможен.
4. Бизнес-ценность системы — в контроле официальных выдач, лимитов и revoke, а не в полной криптографической защите от пересылки.
5. Лимит `5` означает:
   - максимум 5 успешных активаций через бота
   - а не гарантию, что credential физически будет использоваться только 5 живыми людьми

---

## Рекомендуемый MVP scope

Для MVP агент должен реализовать:

- регистрацию Telegram-пользователя
- создание share link
- активацию по `/start <token>`
- лимит `max_uses = 5`
- транзакционную проверку лимита
- **выдачу credentials в Group mode** (один общий credential на шаринг-ссылку)
- уведомление владельца о превышении лимита
- revoke share link
- повторную выдачу уже активированному пользователю без занятия нового слота (тот же групповой credential)

Агент не должен включать в MVP без явной команды:

- Redis
- event sourcing
- микросервисы
- сложную админку
- автоанализ IP
- антифрод
- геоаналитику
- billing
- реферальные уровни
- многоуровневые роли

---

## Будущие расширения

Допускается дальнейшее развитие в сторону:

- **per-user credentials** (режим B), если понадобится точечный revoke и аудит по пользователю
- TTL для credentials
- rotation credentials
- suspicious usage detection
- IP heuristics
- concurrent connection limits
- audit logs
- admin panel

Но агент не должен проектировать MVP так, будто все эти функции уже обязательны.

## Критерий приемки

Система считается реализованной корректно, если выполняются следующие сценарии:

### Сценарий 1
- пользователь A создает ссылку с лимитом 5
- пользователи B, C, D, E, F активируют ее
- все пятеро получают доступ
- в БД ровно 5 активаций

### Сценарий 2
- пользователь G пытается активировать ту же ссылку
- доступ не выдается
- новая активация не создается
- владельцу приходит уведомление о превышении лимита

### Сценарий 3
- пользователь B повторно открывает ссылку
- новый слот не расходуется
- выдается уже существующий доступ или повторно отображается его credential

### Сценарий 4
- владелец отзывает ссылку
- новые пользователи больше не могут пройти активацию

---

## Короткая формула системы

Система должна реализовывать правило:

**Одна шаринг-ссылка позволяет получить доступ только первым 5 пользователям, пришедшим через бота. Шестой пользователь не получает credential.**
